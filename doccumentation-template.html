Project Echo System Architecture (v2.1)

Detailed documentation covering the microservices, data flow, and design decisions for the Echo real-time communication platform.

1. Overview & Goals

Project Echo is designed to be a highly available, low-latency, real-time message broadcasting system. The primary goal was to achieve sub-100ms message delivery reliability using a serverless architecture based on Google Cloud Functions and Firestore.

2. System Architecture

The system utilizes a microservices approach to separate concerns for Authentication, Messaging, and Presence. This modularity allows for independent scaling of each component based on load requirements.

2.1. Component Breakdown

  • **Auth Service:** Handles user login and custom token generation using Firebase Auth.
  • **Message Service:** Manages writing messages to Firestore and triggering fan-out via Cloud Functions.
  • **Presence Service:** Utilizes Firestore's real-time listeners for user online status updates.

3. Data Modeling (Firestore)

All persistent data is stored in **Firestore**. We adopted a denormalized structure to minimize read operations for common queries, particularly for retrieving recent messages.

3.1. Messages Collection Schema

The main collection, /artifacts/{appId}/public/data/messages, stores message documents with the following fields:

{
  "senderId": "string (UID)",
  "text": "string (message content)",
  "timestamp": "timestamp (server time)",
  "roomId": "string (group ID)",
  "readBy": "array (list of user IDs who read it)"
}

4. Client API Endpoints

The client interacts via two primary mechanisms: **HTTPS Callable Functions** for non-realtime operations (like login) and **Firestore Snapshots** for real-time data flow.

4.1. HTTPS Callable: createRoom

This function creates a new chat room and initializes the Firestore document. It requires authentication.

const roomData = await firebase.functions().httpsCallable('createRoom')({ 
  name: 'New Discussion Group',
  type: 'public' 
});
// Response: { roomId: "xyz-123" }

5. Core Design Decisions

The decision to use **Firestore Realtime Listeners** over dedicated WebSocket servers was based on reducing infrastructure maintenance overhead and leveraging the built-in scalability of the platform. While this increases client-side complexity (handling the denormalized data), it drastically simplifies the server-side deployment model.

6. Deployment & Scaling

The entire application is deployed using the Firebase CLI. Scaling is handled automatically by Google Cloud Functions (vertical scaling) and Firestore (horizontal scaling).
